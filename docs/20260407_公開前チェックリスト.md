# 石垣市議会版 みらい議会 公開前チェックリスト

このチェックリストは、石垣市議会版みらい議会を「一般公開しても大丈夫な状態」にするための確認項目をまとめたものです。

## 1. 本番インフラ・デプロイ

- [ ] `web` の Vercel project が本番運用できる状態になっている
- [ ] `admin` を本番公開する場合は、Vercel project を作成済み
- [ ] `web/vercel.json` で Git deployment が有効
- [ ] `admin/vercel.json` で Git deployment が有効
- [ ] GitHub Actions の `Code Check` が成功する
- [ ] GitHub Actions の `Integration Tests` が成功する
- [ ] GitHub Actions の `Migrate DB then Deploy` が成功する
- [ ] `npx supabase migration list --linked` で local / remote の migration history が揃っている
- [ ] `npx supabase migration up --linked --include-all` がエラーなく完了する

## 3. ドメイン・公開設定

- [ ] 独自ドメインを設定する
- [ ] `web` の公開URLを決定する
- [ ] `admin` を外部公開するか、限定公開にするか決める
- [ ] Basic認証を使うかどうか決める
- [ ] 使う場合は `BASIC_AUTH_USER` / `BASIC_AUTH_PASSWORD` を設定する
- [ ] 使わない場合は Basic認証関連 env を未設定にする

## 4. データ整備

### 議案

- [ ] 主要な議案に `bill_contents` を登録する
- [ ] 少なくともトップに出したい議案には `difficulty_level = normal` のコンテンツを入れる
- [ ] 必要なら `hard` のコンテンツも追加する
- [ ] `publish_status` を公開方針に合わせて整える
- [ ] `document_type` が適切に入っている

### トップページ表示

- [ ] 注目表示したい議案に `is_featured = true` を付ける
- [ ] タグで表示したい議案には `tags` と `bills_tags` を設定する
- [ ] トップに出したいタグには `featured_priority` を入れる
- [ ] Coming Soon に出したい議案は `publish_status = coming_soon` にする

### 会期

- [ ] `diet_sessions.is_active` が現在の運用に合っている
- [ ] `diet_sessions.start_date / end_date` が正しい
- [ ] トップ上部の「開催中」表示が期待通りになる日付設定になっている

### 議員名簿

- [ ] `members` テーブルに必要な議員データが入っている
- [ ] 名前、ふりがな、政党、会派、当選回数、生年月日、住所が正しく入っている

## 5. 画面チェック

### 公開サイト

- [ ] トップページが開く
- [ ] 議案詳細ページが開く
- [ ] 議員名簿ページ `/members` が開く
- [ ] 404ページが自然に表示される
- [ ] モバイル表示で崩れない
- [ ] タブレット表示で崩れない
- [ ] OGP / title / description が意図どおり

### 管理画面

- [ ] ログインできる
- [ ] 議案の新規作成ができる
- [ ] `bill_contents` の編集ができる
- [ ] 議案種別の選択ができる
- [ ] ステータス選択が種別ごとに切り替わる
- [ ] 提出議会が固定で正しく入る

## 6. 文言・見た目

- [ ] `国会` や `衆議院` など旧文言が残っていない
- [ ] `法案` が残っていない
- [ ] 石垣市議会向けとして不自然なラベルが残っていない
- [ ] 外部リンクが石垣市議会向けになっている
- [ ] ロゴ非表示の状態が自然に見える
- [ ] ヒーロー画像やトップ構成が意図した内容になっている

## 7. セキュリティ

- [ ] デバッグ用 API や bypass が本番コードに残っていない
- [ ] service role key をクライアント側で使っていない
- [ ] RLS の前提と service role 利用箇所を把握している
- [ ] 不要な公開 endpoint がない
- [ ] `admin` のアクセス制御方針を決めている

## 8. 運用ルール

- [ ] 議案追加の手順を文章にする
- [ ] データ更新担当者を決める
- [ ] 本番更新の流れを決める
- [ ] 障害時の確認先を決める
- [ ] バックアップ方法を決める

## 9. 公開後すぐやること

- [ ] アクセス解析を入れる
- [ ] エラー監視を入れる
- [ ] 週次でバックアップを取る
- [ ] 問い合わせ導線を案内する
- [ ] 最初の数日間は Vercel logs / Supabase logs を重点確認する

## 10. 最終公開判定

以下がすべて満たせたら公開可能と判断する。

- [ ] 本番URLでトップページが安定して表示される
- [ ] 管理画面から議案追加・編集・公開ができる
- [ ] トップに見せたい議案が正しく出ている
- [ ] 一般ユーザーが見る主要ページに致命的な崩れがない
- [ ] CI/CD が成功し、再デプロイ可能な状態である
