## 進め方

### 1. 開発用 DB で確認

まず開発用 Supabase に Topics 関連 migration を反映して、admin / web の表示と運用を確認する。

注意:

- `pnpm db:migrate` はローカル Supabase 向け
- 開発用リモート Supabase に当てるときは `pnpm db:migrate:linked` を使う
- 反映前の確認は `pnpm db:migration:list:linked`

確認ポイント:

- `/topics` が表示される
- `/topics/ishigaki-old-city-hall` が表示される
- admin の `Topics管理` で Topic 作成・更新履歴追加・関連議案追加ができる

### 2. 本番反映 SQL を使う場面

本番用 Supabase は migration を直接流す代わりに、必要に応じて SQL Editor で実行する運用になっているため、Topics も同じ方針で反映する。

本番反映用 SQL:

- `docs/20260415_Topics_本番反映SQL.sql`

### 3. 本番反映後の確認

- 公開サイト `/topics`
- 公開サイト `/topics/ishigaki-old-city-hall`
- 必要なら `/api/revalidate` を実行

revalidate コマンド例:

- ローカル向け Topics のみ: `pnpm revalidate topics`
- 全タグ: `pnpm revalidate --all`
- 公開サイト向け Topics のみ:
  `pnpm revalidate --url https://ishigaki-gikai-app-web.vercel.app topics`
- `curl` で直接実行する場合:
  `curl -X POST "https://ishigaki-gikai-app-web.vercel.app/api/revalidate" -H "Content-Type: application/json" -H "Authorization: Bearer あなたのREVALIDATE_SECRET" -d '{"tags":["topics"]}'`

## 補足

- `20260415113000_seed_topic_history_for_old_city_hall.sql` は初期トピックと初期更新履歴を入れるための seed を含む
- 既に同名 title の `topic_updates` が存在する場合は重複挿入されない
- `topic_bills(topic_id, bill_id)` には unique index を追加しているため、同じ議案を二重に紐付けない
- 現在の `.env` の `REVALIDATE_SECRET` はプレースホルダーなので、そのままでは公開サイト revalidate には使えない
