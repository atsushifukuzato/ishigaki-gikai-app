# 石垣市議会版みらい議会 議案追加運用メモ

## このメモの目的

- 別スレッドに移っても、石垣市議会版の議案追加作業の前提をすぐ共有できるようにする
- 実議案を追加したあと、どの条件がそろえばトップページに出るのかを迷わないようにする
- 議案追加のたびに、ユーザーへ案内するコマンドを定型化する

## 前提

- 対象リポジトリ: `/Users/fukuzatoatsushi/Desktop/ishigaki-gikai/mirai-gikai`
- 公開サイトは `web` のみ
- 本番URL: `https://ishigaki-gikai-app-web.vercel.app`
- 石垣市議会の実議案は `seed` ではなく、本番Supabase向けSQLで管理する
- `packages/seed` は触らない

## 実議案追加の基本方針

- 本文は `public.bill_contents` に入れる
- 議案の基本情報は `public.bills`
- トップ表示は `public.bills_tags` と `public.tags.featured_priority` が重要
- 必要に応じて `public.bills.is_featured` を使うが、石垣市議会版はジャンル別表示を優先
- 今後は、議案本文整備SQLを作った回では、トップ表示用SQLも同じ回で必ず用意する
- つまり「詳細ページで見える」だけでは完了にせず、「トップ表示に必要な設定SQLまである」状態を作業完了の基準にする

## 毎回の完了条件

1. `bill_contents` に `normal` がある
2. `bill_contents` に `hard` がある
3. トップ表示用の `bills_tags` / `tags.featured_priority` 設定SQLがある
4. `revalidate` コマンドまで案内済みである
5. `git add` 対象ファイルを毎回ユーザーへ個別指定している

## トップページに出る条件

議案がトップページのジャンル別表示に出るには、主に次が必要。

1. `bills.publish_status = 'published'`
2. 対応する `bill_contents` がある
   - 通常表示用: `difficulty_level = 'normal'`
   - 詳細表示用: `difficulty_level = 'hard'`
3. `bills.diet_session_id` が active な会期に紐づいている
4. `bills_tags` でタグに紐づいている
5. そのタグに `featured_priority` が入っている
6. キャッシュ更新後の画面確認が済んでいる

## 実務上の確認順

1. Supabase で `bills` / `bill_contents` / `bills_tags` / `tags` を確認
2. 議案詳細ページ直URLで本文が見えるか確認
3. 本文整備SQLとは別に、トップ表示用タグ設定SQLも必ず作る
4. トップ表示用タグ設定を入れる
5. `revalidate` を実行
6. トップページをハードリロードして確認

## 毎回作るファイル

議案追加を1回進めるときは、原則として次をセットで作る。

1. `議案本文整備第N弾SQL.sql`
2. 必要なら `hard追加入力SQL.sql`
3. `第N弾議案_トップ表示追加SQL.sql`
4. `整備状況メモ.md` の更新

トップ表示用SQLは、本文整備後に別途依頼されるのを待たず、最初から作っておく。

## 議案追加後に毎回ユーザーへ案内するコマンド

案内順も固定する。

1. 本文整備SQLを Supabase で実行
2. トップ表示追加SQLを Supabase で実行
3. `revalidate` を実行
4. `git status --short`
5. 今回分だけ `git add`
6. 必要なら `git commit`
7. 必要なら `git push`

### 1. 本番キャッシュ更新

基本はこれを案内する。

```bash
curl -X POST "https://ishigaki-gikai-app-web.vercel.app/api/revalidate" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer あなたのREVALIDATE_SECRET" \
  -d '{"tags":["bills","diet-sessions","interview-configs"]}'
```

### 2. Git の変更確認

```bash
git status --short
```

### 3. 今回分だけ add

`git add .` は使わず、今回追加した `docs/*.sql` と必要なメモだけを個別指定する。

例:

```bash
git add \
  docs/20260410_第3回定例会_議案本文整備第5弾SQL.sql \
  docs/20260410_第3回定例会_議案本文整備第5弾_hard追加入力SQL.sql \
  docs/20260410_第3回定例会_第5弾議案_トップ表示追加SQL.sql \
  docs/20260410_第3回定例会_整備状況メモ.md
```

トップ表示追加SQLを忘れないこと。本文SQLだけ `git add` して終わりにしない。

### 4. staged 内容確認

```bash
git diff --cached --name-only
```

### 5. コミット

```bash
git commit -m "Add Ishigaki R8 third regular session bill SQL batch 5"
```

### 6. 必要なら push

```bash
git push origin main
```

## 別スレッドで共有すべき要点

別スレッドに移ったら、まずこのメモを見る前提で進める。

- `docs/20260410_石垣市議会版_議案追加運用メモ.md`
- その時点で最新の `docs/20260410_*` など、直近の石垣市議会向けSQL
- 議案追加は毎回「本文整備SQL」と「トップ表示追加SQL」をセットで作る

## 補足

- 詳細ページで見えていてトップに出ないときは、まずタグ・会期・featured_priority を疑う
- `revalidate` を実行しても反映がワンテンポ遅れることはある
- ただし、詳細ページで見えるなら DB 反映自体は成功している可能性が高い
- 今後はこちらから、本文追加後にトップ表示用SQLまで自動で続けて用意する
